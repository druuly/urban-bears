# Data Model

Firestore is the single source of truth for the research journal. This
doc lists every collection, the shape of each document, and a quick
read of the security rules.

## Collection tree

```
authors/
  {uid}
articles/
  {articleId}
    likes/
      {uid}
    reposts/
      {uid}
```

## authors/{uid}

Allowlist of users who can publish. Managed manually from the Firebase
console (see [firebase-setup.md](./firebase-setup.md)). The document
ID is the Firebase Auth UID.

| Field | Type | Required | Notes |
| --- | --- | --- | --- |
| `name` | string | no | Display name, informational only. The page reads `displayName` straight off the auth user, not this field. |
| `role` | string | no | Informational (e.g. "Editor"). |

Anything else you add is ignored by the client. The mere **existence**
of `authors/{uid}` is what grants author privileges.

## articles/{articleId}

The main content document. Auto-IDed via `addDoc`.

| Field | Type | Set by | Notes |
| --- | --- | --- | --- |
| `title` | string | composer | Required, plain text. |
| `subtitle` | string | composer | Optional hook line. |
| `body` | string (HTML) | composer | Sanitized via DOMPurify on **both** write and read. Generated by Quill; may contain `<img src="data:…">` inline images. |
| `coverUrl` | string | composer | Either empty or a `data:image/...;base64,...` URL. Downscaled to ≤ 1400 px wide before save. |
| `tags` | string[] | composer | Free-form tags. Used by Most Popular grouping and (legacy) tag chips. |
| `authorUid` | string | composer | Always `request.auth.uid`; enforced by rules. |
| `authorName` | string | composer | Snapshot of `user.displayName` at publish time. |
| `authorPhoto` | string | composer | Snapshot of `user.photoURL` at publish time. |
| `createdAt` | Timestamp | composer | `serverTimestamp()`. |
| `updatedAt` | Timestamp | composer | `serverTimestamp()`. |
| `likeCount` | number | engagement | Incremented/decremented atomically with the corresponding `likes/{uid}` doc. |
| `repostCount` | number | engagement | Same pattern as likes. |
| `viewCount` | number | reader | Bumped once per `openReader()` call. Best-effort; ignore failures. |

### Doc-size budget

Firestore caps a doc at ~1 MiB. The composer enforces a soft cap of
**~850 KB total** before allowing publish to give breathing room. The
estimate is `(body.length + coverUrl.length) * 0.75` (base64 → byte
adjustment). Authors get a toast if they exceed it and need to
remove/shrink images.

Sizing tips that already happen automatically:

- Cover image: re-encoded JPEG at `maxWidth 1400`, quality `0.82`.
- Inline images: re-encoded JPEG at `maxWidth 1100`, quality `0.82`.
- PNGs/GIFs keep PNG to preserve transparency.

## articles/{id}/likes/{uid}

Existence = "this user has liked this article". One doc per user per
article.

| Field | Type | Notes |
| --- | --- | --- |
| `createdAt` | Timestamp | `serverTimestamp()`. |

## articles/{id}/reposts/{uid}

Identical shape to `likes`.

## Security rules — quick read

Source: `firestore.rules`.

```
authors/{uid}         read: anyone;            write: nobody (console only)
articles/{id}         read: anyone;
                      create: signed in
                              AND authors/{uid} exists
                              AND authorUid == request.auth.uid;
                      update: own article (any field)
                              OR diff is exactly likeCount/repostCount/viewCount;
                      delete: own article;
articles/{id}/likes/{uid}   read: anyone; write: same uid only
articles/{id}/reposts/{uid} read: anyone; write: same uid only
```

The "diff only affects counter fields" clause is what lets readers
increment counters without being authors. If you ever add a new
counter field, update that `hasOnly([...])` list in `firestore.rules`
or the increment write will be denied.

## Query patterns used by the client

| Where | Query |
| --- | --- |
| `loadArticles` | `query(articles, orderBy('createdAt','desc'), limit(60))` — single fetch, sorted, capped at 60. Powers both home and archive after client-side reshuffle. |
| `openReader` | `onSnapshot(doc(articles, id))` — live binding for engagement counters while a reader is open. |
| Engagement toggle | `getDoc` → `deleteDoc`/`setDoc` on the `likes`/`reposts` subdoc, then `updateDoc` with `increment(±1)` on the parent. |

If `state.articles.length` ever needs to exceed 60, raise the `limit`
or paginate with `startAfter(lastDoc.createdAt)`.
